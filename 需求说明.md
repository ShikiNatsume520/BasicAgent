# BasicAgent 需求说明

## 一、项目愿景

借鉴 Claude Code 的设计，实现一个基础的 Agent 框架。该框架对外提供封装好的、具有记忆和工具调用功能的 Agent。框架本身不关心具体业务，上层应用（如游戏对话、研究助手等）基于此框架构建专属功能。

### 长期架构

```
┌──────────────────────────────────────────────────────┐
│  上层应用 (游戏对话 / 研究助手 / 自定义Agent)          │
├──────────────────────────────────────────────────────┤
│  QueryEngine (会话管理器)                              │
│  - 创建/删除/恢复 会话                                 │
│  - 会话级工具配置                                      │
│  - 模式管理 (Normal / Plan)                            │
├──────────────────────────────────────────────────────┤
│  QueryLoop (单次 ReAct 循环)                           │
│  - 流式输入/输出                                       │
│  - 工具调用与执行                                      │
│  - 子Agent (skillTool) Fork 与结果回流                 │
├──────────────────────────────────────────────────────┤
│  Context Manager (上下文管理)                          │
│  - 分层上下文: SystemPrompt / Messages[] / Tools[]     │
│  - 三层压缩: snip → microcompact → [collapse|compact|auto]
│  - Memory 注入: 用户约束 → MEMORY索引 → 记忆检索       │
├──────────────────────────────────────────────────────┤
│  Memory System (记忆系统)                              │
│  - 用户约束 (多层级: 项目级 / 用户级)                  │
│  - MEMORY.md + Topic Files (项目级，Agent 自主写入)    │
│  - Session Memory (会话级，压缩产物)                   │
├──────────────────────────────────────────────────────┤
│  Provider Layer (LiteLLM 统一接口)                     │
│  - Anthropic / OpenAI / Google 原生协议                 │
│  - 其余厂商 OpenAI 兼容协议                            │
│  - 流式 + 非流式调用                                   │

│  - 内置工具: Read, Edit, Grep, Bash ...                │

```

## 二、各模块详细设计（完整愿景）

### 2.1 Provider Layer

- 使用 LiteLLM 统一不同供应商的接口
- 必须支持流式调用（streaming）
- 流式场景下需要正确处理：
  - 纯文本流式输出（无工具调用的场景，如游戏对话）
  - 流式工具调用（streaming tool_calls 的缓冲与拼接）
- 对外暴露统一的 message 格式

### 2.2 QueryEngine（会话管理器）

QueryEngine 是全局单例管理器，负责会话生命周期。

#### 接口定义

```
create_session(config) → session_id
  config:
    - tools: list[str]            # 可用工具列表（默认全部内置工具）
    - disabled_tools: list[str]   # 禁用的工具列表
    - system_prompt: str?         # 可选的系统身份提示
    - user_constraints: str?      # 可选的用户约束（项目级，覆盖默认）
    - model: str                  # 模型标识
    - provider: str               # 供应商标识
    - compression_config: dict?   # 压缩策略配置

send(session_id, message) → AsyncIterator[ResponseChunk]
  - 流式返回，支持 text / tool_call / status 等事件类型
  - 内部触发一次完整的 QueryLoop

resume(session_id) → Session
  - 从持久化存储恢复会话上下文

delete(session_id)
  - 删除会话及其持久化数据

list_sessions() → list[SessionMeta]
```

### 2.3 QueryLoop（单次 ReAct 循环）

一次 QueryLoop 是一个完整的工具调用循环：

```
[用户输入] → LLM → [tool_use?] → 执行工具 → tool_result → LLM → ... → [纯文本响应]
```

#### 关键行为

- 支持流式输入：边接收用户输入边传递给 LLM（降低首 token 延迟）
- 支持流式输出：LLM 的 text delta 实时推送给调用方
- 流式工具调用：缓冲 streaming tool_calls 直到参数完整，然后执行
- 循环终止条件：LLM 返回纯文本（无 tool_use），或达到最大工具调用次数
- 子Agent（通过 skillTool 触发）在独立的 QueryLoop 中执行，结果回流主 Agent

### 2.4 Tool System（工具系统）

#### 工具注册与发现

- 所有工具（内置、MCP、skill）统一注册到 ToolRegistry
- 每个工具有唯一的 name、description、input_schema（JSON Schema）
- MCP 连接后自动拉取所有可用工具，注册到 Registry

#### 会话级工具控制

创建会话时通过 `tools` 或 `disabled_tools` 控制该会话可用的工具集。
传给 LLM 的 tools[] 数组只包含当前会话启用的工具。

#### toolSearch（延迟工具发现）

- 这是一个内置工具，LLM 可调用它来获取指定工具的详细 schema
- 主要用于 MCP 工具：MCP 工具注册时只存 name + 简短 description，不注入完整 schema 到上下文
- LLM 调用 toolSearch(query) → 返回匹配工具的完整 input_schema
- 好处：避免大量 MCP 工具的完整 schema 占满上下文

#### skillTool（Skill 加载与子Agent Fork）

- Skill = 一段结构化提示词（prompt template），存储为独立文件
- LLM 调用 skillTool(skill_name, args) 时：
  1. 从 Registry 查找对应 Skill 的提示词
  2. Fork 一个子Agent（独立的 QueryLoop，独立的上下文）
  3. 子Agent 执行 Skill 指令
  4. 子Agent 的最终结果回流到主Agent 的上下文
- 子Agent 与主Agent 的记忆、会话、上下文完全独立

#### 内置工具

| 工具 | 说明 | 备注 |
|------|------|------|
| Read | 读取文件 | |
| Edit | 精确字符串匹配编辑 | 需要 old_string → new_string 的可靠匹配 |
| Grep | 正则内容搜索 | 底层实现待定（ripgrep / 纯 Python） |
| Bash | 执行 shell 命令 | 需要超时控制 |
| toolSearch | 延迟工具发现 | 查询 MCP 工具 schema |
| skillTool | 加载 Skill 到子Agent | Fork + 结果回流 |

#### MCP 工具

- 通过 MCP 协议连接外部工具服务（支持 stdio / SSE 传输）
- 连接后自动发现并注册所有工具
- MCP 工具默认通过 toolSearch 延迟发现（不直接注入完整 schema）
- MCP 工具的启用/禁用与内置工具一致，受会话级工具配置控制

### 2.5 Context Manager（上下文管理）

#### 上下文结构

```
Context Window
├── SystemPrompt          (固定，不可压缩)
├── Messages[]            (动态增长，受压缩管理)
│   ├── [0] 用户约束 + MEMORY.md 索引  (会话开始时注入)
│   ├── [1] 用户第一条消息
│   ├── [2] Assistant 响应 (可能包含 tool_use)
│   ├── [3] User (tool_result)
│   ├── ...
│   └── [末尾] 记忆检索结果 (即时注入)
└── Tools[]               (当前会话启用的工具 schema)
```

**关键规则：**
- 用户约束和 MEMORY.md 索引作为第一条 Message（role=user）注入
- 后台记忆检索的结果即时注入到 Messages[] 末尾
- Tools[] 是当前会话启用的工具的完整 schema 列表
- SystemPrompt 不参与压缩

#### Messages 的 JSON 结构

```
// Assistant 消息（可能包含 tool_use）
{
  "role": "assistant",
  "content": [
    {"type": "text", "text": "..."},
    {"type": "tool_use", "id": "call_xxx", "name": "Read", "input": {...}}
  ]
}

// User 消息（可能包含 tool_result）
{
  "role": "user",
  "content": [
    {"type": "tool_result", "tool_use_id": "call_xxx", "content": "..."}
  ]
}
```

**重要约束：** tool_use 与 tool_result 必须一一匹配。压缩机制必须维护这个不变量。

### 2.6 三层压缩机制

#### 第一层：Snip（滑动窗口）

- **触发条件：** Messages[] 的 token 总量超过滑动窗口大小
- **行为：** 将窗口外的远古消息移出上下文
- **窗口大小：** 按 token 计算，可配置
- **注意：** 可能破坏 tool_use / tool_result 的配对关系，由第二层修复

#### 第二层：Microcompact（微压缩）

两个功能：
1. **工具结果裁剪：** 将过长的 tool_result 进行截断或摘要
2. **配对修复：** 检查并修复 snip 可能造成的 tool_use 与 tool_result 不匹配
   - 孤立的 tool_result（对应 tool_use 已被移除）→ 删除该 tool_result
   - 或者为孤立的 tool_result 构造一个占位 tool_use

#### 第三层：深度压缩（三种模式，超参数控制启用）

**模式 A：Context Collapse（上下文折叠）**

- 后台 Fork 一个子Agent，对 Messages[] 进行分组标记
- 将 Messages[] 划分为若干大组（如：按话题、按工具调用批次）
- 将部分组替换为 **摘要 + 指针**
- **无损压缩：** 指针指向持久化文件，LLM 可通过 Read 工具读取原始内容
- 适用场景：需要保留上下文关联但减少 token 占用

**模式 B：Session Compact（会话压缩）**

- 固定唤醒一个 Fork 子Agent，从当前对话中提取关键信息写入 Session Memory
- 使用 Session Memory 的内容替换掉较早的对话历史
- **有损压缩：** 原始对话细节丢失，只保留提取的摘要

**模式 C：AutoCompact（自动压缩）** — 默认启用

- 将选定的历史对话传给 LLM，返回总结
- 用总结替换原始对话
- **有损压缩：** 最简单直接的压缩方式

#### 压缩配置

```python
compression_config = {
    "snip": {
        "window_tokens": 100000,
    },
    "microcompact": {
        "max_tool_result_tokens": 5000,
    },
    "tier3": {
        "mode": "auto",       # "collapse" | "session" | "auto"
        "threshold": 0.8,     # 触发第三层压缩的 context 占用比例
    }
}
```

### 2.7 Memory System（记忆系统）

#### 用户约束（User Constraints）

- 多层级：项目级 + 用户级
- 项目级：存储在项目目录下（如 `./CLAUDE.md`）
- 用户级：存储在用户目录下（如 `~/.basic-agent/constraints.md`）
- 加载顺序：用户级 → 项目级（项目级优先级更高）
- 注入方式：作为 Messages[] 的第一条 user message

#### MEMORY.md（项目级 Agent 记忆）

- 索引文件：`[Title](file.md) — one-line hook` 格式
- 索引常驻：MEMORY.md 索引在会话开始时注入上下文（作为第一条 message 的一部分）
- Topic Files：每条索引对应一个独立的 .md 文件，存储详细内容
- 记忆检索：后台子Agent 根据用户请求和索引，检索相关的 Topic File 内容，即时注入 Messages[] 末尾
- 记忆写入：每轮 QueryLoop 结束末尾，扫描新内容，尝试提取相关信息并写入（MEMORY.md 是项目级的，记录用户行为特征、对回答的反馈等）
- MEMORY.md 的写入会影响后续新会话，不影响当前会话（前缀稳定性考虑）

#### Session Memory（会话级记忆）

- 来源：压缩机制的产物（Session Compact / AutoCompact 生成的摘要）
- 用途：作为后续压缩的参考，或在会话恢复时提供上下文
- 存储：随会话一起持久化

### 2.8 持久化

- 会话消息历史：JSON 文件
- 用户约束：md 文件
- MEMORY.md + topic files：md 文件
- Session Memory：随会话 JSON 存储

### 2.9 扩展机制

- **MCP：** 通过 MCP 协议连接外部工具服务
- **Plugin：** 打包 "/" 命令 + subAgent + Skill + MCP 的安装接口
- **"/" 命令系统：** 用户可注册自定义命令（v2）
- **模式切换：** Normal mode / Plan mode（v2）

v1.0 的首要目标：**支撑后续 LLM 对话游戏项目**。

游戏端使用 Unity 制作，通过 HTTP 与后端通信。v1.0 交付物包含一个可 Docker 化的后端服务。
